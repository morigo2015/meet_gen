# Технічне завдання: система генерації Word-документів з Excel-даних

## 1. Мета

Розробити Python-систему для автоматичної генерації набору Word-документів (`.docx`) на основі:

- одного Word-шаблону (`.docx`);
- одного Excel-файлу (`.xlsx`) з таблицею даних;
- правила: **один рядок Excel = один згенерований Word-файл**.

У Word-шаблоні використовуються placeholder-и, які замінюються значеннями з відповідного рядка Excel.

---

## 2. Вхідні та вихідні дані

### Вхідні дані

```text
template.docx
Дані.xlsx
output_directory/
```

Excel-файл містить таблицю, де:

- перший рядок — назви колонок;
- кожен наступний непорожній рядок — дані для окремого документа;
- назви колонок можуть бути латиницею або кирилицею;
- всі текстові дані мають підтримувати UTF-8.

Приклад Excel:

```text
номер_договору | назва_клієнта | дата_договору | сума
001            | ТОВ Альфа     | 01.05.2026    | 10000
002            | ТОВ Бета      | 02.05.2026    | 25000
```

### Word-шаблон

У Word-шаблоні використовуються placeholder-и такого формату:

```text
{{ номер_договору }}
{{ назва_клієнта }}
{{ дата_договору }}
{{ сума }}
```

### Вихідні дані

Для кожного валідного рядка Excel система створює окремий `.docx` файл у вихідній директорії.

Приклад:

```text
output/generated/001_ТОВ_Альфа.docx
output/generated/002_ТОВ_Бета.docx
```

---

## 3. Важливі припущення та обмеження MVP

У першій версії системи приймаються такі обмеження:

1. Кожен placeholder повністю знаходиться в одному Word `run` у межах одного `paragraph`.
2. Placeholder-и не розбиті між різними `run`, `paragraph`, таблицями або іншими частинами документа.
3. Підтримується тільки проста заміна placeholder-ів у форматі:

```text
{{ назва_колонки }}
```

4. У MVP не підтримуються:

```text
{{ client.name }}
{{ amount | currency }}
{% if condition %}
{% for item in items %}
```

5. Назви placeholder-ів мають точно відповідати назвам колонок Excel після нормалізації пробілів.
6. Всі placeholder-и, які використовуються в шаблоні, вважаються обов’язковими полями.
7. Система не змінює оригінальний Word-шаблон.
8. Система не потребує встановленого Microsoft Word.
9. Усі тексти, назви колонок, значення полів, імена файлів і повідомлення про помилки мають підтримувати кирилицю та UTF-8.

## 4. Технологічний стек

Основний стек MVP:

```text
Python 3.11+
docxtpl
openpyxl
pydantic
typer

```

Призначення бібліотек:

- `docxtpl` — рендеринг `.docx` шаблону;
- `openpyxl` — читання Excel-файлу `.xlsx`;

## 6. Основні модулі

### 6.1. `excel_reader.py`

Відповідає за читання Excel-файлу.

Вимоги:

- читати `.xlsx` через `openpyxl`;
- за замовчуванням використовувати перший sheet;
- підтримувати явне зазначення sheet через CLI-параметр;
- читати перший рядок як назви колонок;
- пропускати повністю порожні рядки;
- повертати кожен рядок як `dict`;
- підтримувати кириличні назви колонок і значення;
- виявляти дублікати назв колонок;
- зберігати номер рядка Excel для звітів про помилки.

Приклад результату:

```python
{
    "_row_number": 2,
    "номер_договору": "001",
    "назва_клієнта": "ТОВ Альфа",
    "дата_договору": "01.05.2026",
    "сума": "10 000"
}
```

---

### 6.2. `template_validator.py`

Відповідає за перевірку відповідності Word-шаблону та Excel-даних.

Вимоги:

- витягнути список placeholder-ів із Word-шаблону;
- перевірити, що кожен placeholder має відповідну колонку в Excel;
- перевірити, що placeholder-и мають коректний формат `{{ назва_колонки }}`;
- перевірити, що для кожного рядка всі поля, які використовуються в шаблоні, мають непорожні значення;
- сформувати список помилок із зазначенням рядка Excel або назви placeholder-а.

Приклад помилки:

```text
У шаблоні використано placeholder {{ назва_клієнта }}, але в Excel немає колонки "назва_клієнта".
```

---

### 6.3. `word_renderer.py`

Відповідає за створення `.docx` файлів.

Вимоги:

- завантажувати Word-шаблон через `docxtpl`;
- для кожного валідного рядка виконувати `render(context)`;
- зберігати окремий `.docx` файл;
- не змінювати оригінальний шаблон;
- максимально зберігати форматування шаблону;
- коректно працювати з UTF-8 і кирилицею.

---

### 6.4. `filename_builder.py`

Відповідає за формування безпечних назв файлів.

Базовий шаблон імені файлу:

```text
{номер_договору}_{назва_клієнта}.docx
```

Вимоги:

- підтримувати кирилицю в назвах файлів;
- замінювати заборонені символи файлової системи на `_`;
- обрізати надто довгі назви файлів;
- якщо значення для назви файлу відсутні, використовувати fallback:

```text
document_0001.docx
document_0002.docx
```

- якщо файл із такою назвою вже існує, додавати суфікс:

```text
001_ТОВ_Альфа_2.docx
001_ТОВ_Альфа_3.docx
```

---

### 6.5. `formatters.py`

Відповідає за нормалізацію значень перед підстановкою в шаблон.

Вимоги MVP:

- дати форматувати як `DD.MM.YYYY`;
- цілі числа виводити без десяткової частини;
- числа з дробовою частиною виводити з двома знаками після коми;
- текстові значення обрізати від зайвих пробілів на початку та в кінці;
- порожні значення для полів, що використовуються в шаблоні, вважати помилкою.

## 7. CLI-інтерфейс

CLI реалізується через `typer`.

Базовий запуск:

```bash
python -m src.main generate \
  --template ./templates/template.docx \
  --data ./input/Дані.xlsx \
  --output ./output/generated
```

Додаткові параметри:

```bash
--sheet "Аркуш1"
--filename-template "{номер_договору}_{назва_клієнта}"
--strict
--dry-run
```

### Режими роботи

#### `strict`

- будь-яка помилка зупиняє генерацію;
- якщо знайдено помилки у шаблоні або Excel, файли не створюються.

#### non-strict

- помилкові рядки пропускаються;
- валідні рядки генеруються;
- у фінальному звіті показується список пропущених рядків і причини.

#### `dry-run`

- файли не створюються;
- система виконує всі перевірки;
- повертає звіт про потенційні помилки та кількість документів, які могли б бути створені.

## 8. Логування та звітність

Після завершення система виводить короткий звіт у консоль:

```text
Processed rows: 100
Generated documents: 98
Skipped rows: 2
Output directory: ./output/generated
```

Для помилок рядків потрібно виводити номер рядка Excel:

```text
Row 15: missing required field "номер_договору"
Row 27: invalid date value in "дата_договору"
```

Рекомендовано створювати log-файл у вихідній директорії:

```text
generation.log
```

## 9. Обробка помилок

Система має коректно обробляти такі помилки:

- Word-шаблон не знайдено;
- Excel-файл не знайдено;
- файл має неправильний формат;
- Excel-файл порожній;
- sheet не знайдено;
- у Excel відсутні заголовки колонок;
- знайдено дублікати назв колонок;
- placeholder у шаблоні не має відповідної колонки в Excel;
- значення обов’язкового поля порожнє;
- неможливо створити вихідну директорію;
- неможливо записати файл;
- згенеровано дублікат назви файлу.

Повідомлення про помилки мають бути зрозумілими та підтримувати кирилицю.

## 10. Основний алгоритм

```text
1. Прочитати CLI-параметри.
2. Перевірити існування template.docx і data.xlsx.
3. Створити output directory, якщо її немає.
4. Прочитати Excel-файл.
5. Отримати список колонок.
6. Витягнути placeholder-и з Word-шаблону.
7. Перевірити відповідність placeholder-ів і колонок.
8. Для кожного непорожнього рядка Excel:
   8.1. сформувати context;
   8.2. нормалізувати текст, дати та числа;
   8.3. перевірити обов’язкові поля;
   8.4. у non-strict режимі пропустити невалідний рядок;
   8.5. у strict режимі зупинити процес при помилці;
   8.6. сформувати безпечну назву файлу;
   8.7. згенерувати .docx через docxtpl;
   8.8. зберегти файл у output directory.
9. Вивести фінальний звіт.
10. Записати log-файл.
```

## 12. Acceptance Criteria

Система вважається готовою, якщо виконуються такі умови:

1. Для Excel-файлу з 10 валідними рядками створюється 10 окремих `.docx` файлів.
2. У кожному згенерованому файлі всі placeholder-и замінені на відповідні значення з Excel.
3. Кириличні назви колонок, значення полів і назви файлів обробляються коректно.
4. Якщо в шаблоні є placeholder без відповідної колонки в Excel, `dry-run` повертає помилку.
5. Якщо в рядку Excel порожнє значення для поля, яке використовується в шаблоні, цей рядок не генерується в non-strict режимі.
6. У `strict` режимі генерація зупиняється при першій помилці, і файли не створюються.
7. У `dry-run` режимі файли не створюються.
8. Оригінальний `template.docx` не змінюється.
9. При дублюванні назв файлів система додає числовий суфікс.
10. Всі unit-тести проходять командою `pytest`.
11. CLI-команда генерації працює через `typer`.
12. README містить інструкцію запуску та приклади вхідних файлів.
